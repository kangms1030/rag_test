## P0-E — VRAM/스왑 실측 `[p0e_vram.json]`

측정값(`ollama ps` 실측 VRAM):

| 모델 | 상주 VRAM | 비고 |
|---|---|---|
| embeddinggemma | 0.68 GB | 100% GPU |
| gemma4:12b (num_ctx 16384) | 8.4 GB | 100% GPU |
| **gemma4:e4b (num_ctx 2048)** | **3.3 GB** | 디스크 9.6GB와 무관 — KV 축소 시 실제 점유는 작음 |
| 12b + e4b + embeddinggemma 공존 | **11.7 GB** | 최종 스냅샷에서 안정적 공존 확인 |

- 스왑 지연: 상주 유지 시 e4b 재호출 load 0.59s(재로드 없음). 단 **최초 e4b 로드 시 Ollama가 12b를 일시 축출**하는 스케줄러 동작이 1회 관측됨 → 직후 12b 재호출에 6.63s 재로드 발생.
- **계획 수정 1 (F5 완화)**: "12b+e4b 동시상주 불가(17.2GB)"는 디스크 크기 기준의 과대추정이었다. **e4b를 num_ctx 2048로 고정하면 온라인 경로에서 12b와 공존 가능(합 ~12.4GB, +리랭커 1.5GB=~13.9GB, 16GB 내)**. 다만 최초 로드 시 축출 리스크가 있으므로, 온라인 검증/판정의 **1차 기본값은 여전히 "상주 중인 12b 짧은 호출"**(스왑 리스크 0)로 두고, e4b 온라인 사용은 워밍업을 controller가 강제(양쪽 사전 로드)하는 옵션으로 채택한다.

## P0-C — 빈 응답/거절 원인 분리 `[p0c_ctx_overflow.json]`

catalog 실패 6건을 원본 answer 파이프라인 그대로 재구성해 num_ctx 8192 vs 16384로 재현:

| qid | 프롬프트 | prompt_eval | 8192 | 16384 | 실제 원인 |
|---|---|---|---|---|---|
| vp_002 | 15,658자 | 7791 | 빈 응답 | **정상** | 프롬프트는 안 잘림(7791<8192). 8192는 생성 여유 부족(출력 굶김) → 16k가 해소 |
| vp_006 | 7,932자 | 3685 | 빈 응답 | 빈 응답 | 오버플로 아님. 표 HTML(6566자)에 모델이 빈 응답 |
| vp_007 | 6,525자 | 3102 | 빈 응답 | **정상** | 16k가 해소(부차 효과/비결정성) |
| vp_010 | 962자 | 585 | 확인불가 | 확인불가 | **figure 페이지(텍스트 158자)를 text 라우팅 → 정직한 거절** |
| vp_014 | 4,652자 | 2312 | 확인불가 | 확인불가 | 동일: figure 페이지 라우팅 결함 |
| vp_005 | (게이트 none) | - | - | - | 문서 후보 없음 |

- **계획 수정 2 (F1/F8 반증)**: 프롬프트 truncation은 **한 건도 발생하지 않았다**(최대 peval 7791). 빈 응답의 실제 원인은 3종으로 갈린다:
  1. **출력 토큰 굶김**(vp_002): 큰 표HTML 컨텍스트로 num_ctx 8192에 생성 여유가 없어 빈 응답. → **num_ctx 16384 상향이 해결**(계획대로 채택, 무해).
  2. **모델 자체 빈 응답**(vp_006): 여유가 충분해도(peval 3685) 특정 스캔+표 HTML에서 빈 출력. → 표 HTML 단순화/전처리 또는 vision 교차가 필요.
  3. **라우팅 결함**(vp_010/014): 정답이 텍스트 빈약한 figure 페이지인데 1순위 기준 text 라우팅 → 근거 부족으로 정직한 거절. → **리랭커+다중페이지 라우팅(Phase 1/2)로 해결**, 계획의 핵심 방향과 일치.
- 즉 "근거 있는데 실패 33%"는 단일 오버플로가 아니라 위 3종 혼합이며, **num_ctx 상향은 일부(1)만 해결**한다. 나머지는 랭킹/라우팅/표전처리가 담당 — 계획의 상류(랭킹) 우선 전략이 유효함이 재확인됨.

## P0-D — 리랭커 선검증 (bge-reranker-v2-m3) `[p0d_rerank.json]`

기존 page_index를 GLOBAL top20 질의 후 리랭크. 36문항(rag_eval 16 + vlm_probe 20). **kiwi BM25**(kiwipiepy 설치 후):

| 지표 | baseline(RRF, kiwi) | reranked | 개선 |
|---|---|---|---|
| page_hit@1 | 0.361 | 0.417 | **+5.6%p** |
| page_hit@3 | **0.556** (test_2 보고치와 정확 일치) | 0.639 | **+8.3%p** |
| page MRR | 0.472 | 0.540 | **+14% 상대** |
| 리랭크 지연(20쌍) | — | 평균 2.55s / 최대 3.6s | 목표 내 |

- 승격 4건(core_002, vp_001/002/003 → 1순위). 훼손 2건(core_013, vp_010 = figure/반복서식 스캔 페이지, 텍스트 빈약).
- **환경 갭 발견**: intern_chatbot 환경에 **kiwipiepy가 미설치**였다(BM25가 char_bigram으로 폴백 중이었음). Phase 0에서 설치 완료. baseline @3가 설치 후 test_2 보고치(0.556)와 정확히 일치 → 재현성 확보.
- **계획 수정 3 (조건부 채택)**: page_hit@3 +8.3%p는 통과 바(+10%p)에 근소 미달이나, 이는 **페이지 단위 리랭킹 + 긴 페이지 truncation(2048토큰)** 이라는 불리한 조건의 결과다. 계획 Phase 1은 **청크 단위 리랭킹(truncation 없음)** 이므로 개선폭이 더 클 것으로 기대. 훼손 2건도 전부 텍스트 빈약 페이지(청크화+표 단독청크로 완화 대상). → **리랭커 채택하되 Phase 1에서 청크 단위로 P0-D 재검증**을 통과 조건으로 명문화. Qwen3-Reranker 교체는 청크 재검증에서도 미달 시로 보류.

리랭커 상주 VRAM(fp16)은 로드 성공 및 정상 추론 확인. 12b(8.4)+embeddinggemma(0.68)+reranker(~1.5) ≈ 10.6GB로 16GB 내 여유.

## P0-A — 해상도/타일링 오독 `[p0a_resolution.json]`

오독 5건 + 대조 1건 × 조건별로 12b에 전사시켜 gold 토큰 포함 & 기지 오독 미포함을 채점:

| 조건 | 전사 정확 | 교정된 케이스 |
|---|---|---|
| page_hidpi350 (PDF 페이지 350DPI 단일 이미지) | 1/6 | vp_011(대조군)만 |
| **page_tiled2x2 (300DPI 페이지를 2×2 타일 4장)** | **3/6** | **vp_009(캐너→개요/갤럭시), vp_010(집선), vp_011** |
| crop_hidpi400 (표 bbox 400DPI 크롭) | 0/2 | 없음 |
| baseline (test_2 실전송) | 0/0 | *프로브 아티팩트: manifest의 stale 이미지 경로로 미실행. fact §4.3B가 이 6건을 baseline 오독으로 확정* |

- **계획 F4 확증**: 해상도만 올린 단일 이미지(page_hidpi350)는 1/6(대조군뿐)로 거의 무효. 반면 **같은 페이지를 2×2 타일 4장으로 개별 전송하면 3/6**로, 실제 오독 2건(vp_009 "캐너", vp_010 "집선/확장")을 교정했다. Ollama가 단일 이미지를 비전 인코더 입력(≈896px/280토큰)으로 뭉개는 문제(#10392 pan&scan 미구현, #15626 max_soft_tokens 하드코딩)를, **타일=이미지당 개별 인코딩(수동 pan&scan)** 으로 우회한 것.
- **한계**: 조밀한 시리얼번호(vp_006 R54KB00TXWF)·작은 숫자(vp_007 135)·단일 용어(vp_014 인증)는 타일링으로도 미교정. 2×2 격자로는 이런 미세 텍스트 해상도가 부족(더 촘촘한 타일이 필요하거나 근본적으로 VLM 한계).
- **Phase 3 방향 확정**: (주) **MinerU 텍스트 경로로 vision 의존 자체를 줄이는 것**이 여전히 1순위(text 경로 숫자정확도 100%). (보조) vision이 불가피할 때만 **2×2(또는 더 세분) 타일링**을 적용하고, 해상도 단순 상향은 채택하지 않는다. 타일링은 VLM 호출 이미지 수를 늘려 지연이 증가하므로(4장) figure/표 라우팅에 한해 선택 적용.

## P0-B — MinerU 2.5 (보류, 환경 정비 필요)

- **발견**: 현재 `intern_chatbot` 환경에 **MinerU 패키지가 설치되어 있지 않다**(pip list에 mineru/magic-pdf 없음). test_2의 파싱 캐시는 다른 환경/시점에서 생성된 것으로 보인다.
- 따라서 P0-B(MinerU 2.5 VLM 백엔드로 스캔 표 재파싱)는 **MinerU 2.5 설치 + 1.2B 모델 다운로드 + (vllm 또는 transformers 백엔드) 구성**이 선행되어야 한다. 이는 Windows에서 실패 리스크가 있는 다단계 설치라, 다른 프로브(E/C/D/A)와 분리해 **Phase 3 준비 단계의 독립 작업으로 보류**한다.
- P0-B는 Phase 3(vision 개선)에만 영향을 주므로 Phase 1/2 착수를 막지 않는다.
- **다음 세션 실행 레시피**(예정): `pip install mineru[core]` 또는 최신 mineru → `opendatalab/MinerU2.5-2509-1.2B` 다운로드 → 오독 7건 페이지만 backend=vlm으로 재파싱해 표 텍스트 추출 정확도 A/B → ≥6/7이면 969p 전체 재파싱 채택.
- **부가 조치 필요**: 런타임 환경(intern_chatbot)에 MinerU를 설치해야 test_3가 신규 문서를 ingest할 수 있다(현재는 캐시된 13문서만 재사용 가능).

## Phase 0가 확정한 계획 수정 사항 (Phase 1 반영)

1. **e4b 온라인 사용 가능**(num_ctx 2048=3.3GB, 12b와 공존). 단 기본값은 12b 겸용 유지, e4b는 controller 워밍업 강제 옵션.
2. **빈 응답은 오버플로가 아님** → num_ctx 16384는 채택(무해, 일부 해결)하되, 주 해결책은 리랭커+다중페이지 라우팅+표HTML 전처리.
3. **리랭커는 청크 단위 재검증을 통과 조건으로 채택**. page_hit@3 +8.3%p(page-level)를 청크-level에서 +10%p 이상으로 끌어올리는 것이 Phase 1 통과 기준.
4. **vision 개선은 타일링만 채택**(해상도 상향 폐기). 단 주 전략은 MinerU 텍스트화로 vision 회피, 타일링은 figure/표 라우팅 잔여분의 보조 수단.
5. **환경 갭 2건 해소/기록**: kiwipiepy 설치 완료(BM25 재현성 확보), MinerU 미설치는 Phase 3 전 정비 필요.
