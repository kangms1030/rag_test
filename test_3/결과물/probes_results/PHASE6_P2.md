## 모델 확정 (계획 대비 변경 — 불가피)

- 계획 지정 `gemini-2.5-flash`는 이 API 키(신규 프로젝트)에 **404 "no longer available to new users"**
  — 모든 variant(`-002/-001/-latest/-preview-*`)·`v1`/`v1beta` 전부 확인. `gemini-2.0-flash` 계열은
  **429 "check your plan"**(무료 불가). ⇒ **사용자 결정으로 `gemini-3.1-flash-lite`(lite tier) 채택**
  (최저가·최고 무료한도·stable). 풀 Flash 대비 사고력↓ 가능성은 실측으로 검증(아래 결과는 오히려 개선).
- 무료 티어 **20 RPM** 실측 → 백엔드에 프로세스 전역 **3.2s 스로틀**(429 예방). raw API 지연은
  스로틀 sleep과 분리 계측(`gemini_api_s`).
- 비용 요율은 **가정치**(lite: in $0.10/M, out $0.40/M) — 토큰 수(usageMetadata)는 정확, 요율은 확인 필요.

## 중대 발견 — verify 계층이 12b 전용이라 Flash에서 오작동

- **증상**: 관련 31문항 중 **20문항이 conf=abstain인데 kw>0**(실제 grounded 내용 존재). machinery
  `answered_rate=0.258`은 **심각한 과소집계** — 내용 기반 답변율은 **0.903(28/31)** vs baseline 0.645.
- **기전**: Flash는 "X는 확인할 수 없으나 근거상 Y입니다"식 **hedged 답변**을 자주 쓴다. `verify.is_abstain()`
  (마커 "확인할 수 없"·"제공된 근거" 등, 12b의 blunt 스타일에 캘리브레이션)이 이 부분답변을
  abstain으로 **오탐** → 매 문항 spurious `rollback_text_top1`(불필요한 3번째 호출) + conf=abstain 오라벨.
- **함의**: (1) Flash의 confidence 태그는 신뢰 불가 → 보고는 **내용 기반 답변율**로. (2) verify/롤백
  머신러리는 **모델 교체 시 재캘리브레이션 필요**. 이 "회피 오탐"은 정확히 §3.2-2의 문제이며
  **P3(d) 문장단위 인용검증**이 처방 — 부분답변을 abstain 처리하지 않고 문장별로 유지/제거.
- 환각 0에는 영향 없음(결정론 숫자대조는 그대로 통과). 다만 호출수·비용이 롤백만큼 증가(avg 2.06 gemini calls).
