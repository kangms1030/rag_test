## 1. 목적과 결론

test_2(MinerU 사전파싱 RAG)는 빠르지만 챗봇 모듈로 쓰기엔 결함이 있었다: **vision 오독 86%,
근거 있어도 실패 33%, page_hit@3 55.6%, 무검증 단방향**. test_3은 기조(MinerU 사전파싱 →
그림/표 처리 → LLM 답변)를 유지하며 **시간·정확도를 동시 개선**했다.

**결론: 계획의 정량목표를 대부분 달성**했고, 챗봇 신뢰성의 핵심(환각 0 · 무관거절 100% ·
정직한 abstain)을 확보했다. 남은 실패는 파이프라인 로직이 아니라 **래스터 표/카드의 파싱 한계**와
**일부 검색 미스**로 좁혀졌다.

## 3. 무엇을 어떻게 바꿨나 (Phase 0~4)

- **Phase 0 (진단)**: 5개 프로브로 계획 가정을 실측 검증/반증. 리랭커 page-level +8%p 확인,
  vision은 Ollama 런타임 한계(F4)로 해상도 무효 확증.
- **Phase 1 (검색 코어)**: 페이지 통짜 → **MinerU 블록 청크 색인**(969p→2417청크) +
  **bge-reranker-v2-m3** + **카탈로그 게이트 제거→메타 프리픽스 주입**. Chroma HNSW 대형 컬렉션
  크로스프로세스 로드 실패(B6)를 **numpy 브루트포스+BM25 flat 인덱스**로 원천 제거.
- **Phase 2 (검증·롤백)**: controller 상태기계 + verify(숫자 결정론 대조/groundedness) +
  judge(CRAG 재검색) + 결정론 롤백 3종. **환각 0**, 근거존재-실패 개선.
- **Phase 3 (vision)**: P0-B로 전략 반전 — MinerU **vlm-engine이 텍스트화하는 건 스캔 표가 아니라
  벡터 도표**임을 발견. **figure 페이지 115/117을 vlm-engine으로 텍스트화 → pipeline 캐시 병합 →
  text 경로**로 답변(vision 회피). vision 오독 86%→0%. 부수 효과로 **검색 앵커가 생겨 page_hit@3이
  0.645→0.774로 상승**(Phase 1 목표를 사후 달성시킨 핵심 요인).
- **Phase 4**: Contextual ablation **미채택**(§6), 평가셋 확장 **보류**(§6), 본 통합 리포트 작성.

## 4. 후반 핵심 발견 — 12b 콜드스타트 변동성 (별도 신뢰성 이슈)

phase3_final에서 core_001/007이 "회귀"로 보였으나, 전수 추적 결과 **Phase 3 탓이 아니라 Gemma의
whitespace 런어웨이**였다:

- **기전**: 특정 프롬프트를 Ollama 서버가 **처음 처리(콜드 KV)하면 답변 도중 공백/개행 토큰을
  num_predict(1024)까지 폭주 생성 → `done_reason=length`로 답변이 잘림**(core_001: 1024토큰
  생성했으나 실질 61자 "100M 월"에서 종료, kw 0.429). **동일 프롬프트 재호출 시 서버 KV가 웜이라
  정상 완성**(done_reason=stop, kw 1.0). 콜드는 결정론적(cold-A==cold-B)이라 일반 워밍업은 무효.
- **수정**: `done_reason=length`면 **동일 호출 1회 재발행**(temp=0 그리디 유지 → 숫자 충실도 보존),
  **문항당 예산 2회 상한**(병리적 표문맥의 지연 폭발 차단). `repeat_penalty`는 숫자 반복 억제
  위험으로 배제. 코드: `models._chat`, config `ollama_retry_on_length`/`ollama_max_length_retries`.
- **효과(36문항 vs phase3_final)**: **avg_kw_hit 0.443→0.528(+8.5%p)**, answered 0.613→0.645,
  **개선 core_001/007/vp_004/vp_012, 회귀 0**. `length_retry`가 **18/36 문항에서 28회** 발동 —
  콜드 폭주가 평가의 **절반을 오염**시키고 있었음을 확증. 상시구동(keep_alive) 챗봇에선 웜이라
  드물지만, **재현 가능한 평가·첫 질의 신뢰성**을 위해 필수 수정.

## 5. 실패 사례 전수 분석 (관련 31문항)

**성공 19 / 검색미스 7 / 근거존재-abstain 5.** 환각은 0 — 모든 실패는 정직한 abstain이다.

### (A) 검색 미스 7 — 정답 페이지가 top3 밖 (Phase 1 잔여)
`core_003, core_005, core_012, vp_005, vp_008, vp_011` (6건) — 반복서식/figure 페이지라 텍스트
앵커가 약해 상위로 못 올림. **6건 모두 정직 abstain**(환각 없음).
`vp_015` — 엄격 페이지매칭상 미스지만 **kw=0.8·정답 생성**(인접/형제 페이지에 답이 있어 실질 성공).

### (B) 근거존재-abstain 5 — 파싱/코퍼스 한계
`vp_006, vp_007`(래스터 UI 스크린샷 표), `vp_013`(카드 UI), `core_002, core_008`(지저분한 표 HTML).
정답 페이지는 top3에 있으나 **MinerU OCR·표HTML이 모델이 읽을 형태로 안 뽑혀** 모델이 abstain.
vlm-engine·pipeline 모두 한계(Phase 3 P0-B에서 확인). **파이프라인 로직이 아닌 파싱 천장.**
→ 계획의 "근거존재-실패 0%"를 완전 달성 못 한 유일 원인(33.3%→20.8%로 개선은 함).

### (C) 성공 19 — kw 0.17~1.0
figure 도표(vp_004/009/010/012/014)가 Phase 3 텍스트화로 정답화된 게 두드러짐. core_001/011,
vp_002/003, vt_001/002 등 완전정답(1.0) 다수.

### 지연
성공 답변은 대개 25~90s. 최대 148.6s는 (B)의 병리적 표 문맥이 재시도 상한(2회)까지 소모한
경우로, **모두 180s 데드라인 내**. 상시 웜 환경에선 재시도가 거의 없어 더 빠름.

## 6. Phase 4 처리 내역 (미채택·보류 기록)

- **Contextual Retrieval ablation — 미채택(사용자 결정)**: 계획상 'page_hit@1 +5%p 이상일 때만
  채택'하는 조건부 항목. **page_hit@3(0.774)이 이미 목표 초과**, 유사 실험(rerank raw_text =
  프리픽스 제거)이 **오히려 회귀**(@1 0.645→0.452), e4b 오프라인 2~5h 비용 대비 이득 불확실 →
  실행하지 않고 조건부 미채택으로 종결. 향후 코퍼스 대폭 확장 시 재검토.
- **평가셋 확장(~50문항) — 보류(사용자 결정)**: 신규 문항 정답은 사람 검수가 필요한 협업 항목.
  기존 36문항 회귀셋으로 최종 리포트를 확정하고, 확장은 별도 세션에서 협업 진행.

## 7. 남은 한계와 향후 과제

1. **래스터 표/카드 파싱 천장**(§5-B): 현행 로컬 스택으론 한계. 필요 시 표 전용 고정밀
   OCR/전사 모델 도입 검토.
2. **검색 미스 6건**(§5-A): 반복서식 페이지의 고유 앵커 강화(지역명/표 헤더 청크화)나 인접
   페이지 확장 검색.
3. **콜드스타트 지연**: 상시 웜(keep_alive)로 대부분 회피되나, 첫 질의는 재시도로 느릴 수 있음.
4. **평가셋 확장**: 홀드아웃 ~14문항으로 소표본 과적합 점검(협업 대기).

## 8. 최종 판정

계획의 5대 정량목표 중 **page_hit@3 · doc_hit · vision 오독률 · 무관거절 · 모델호출**을 달성/초과.
**근거존재-실패율**만 파싱 한계로 0% 미달(33.3%→20.8% 개선). **환각 0·무관거절 100%**의 챗봇
신뢰성 요건을 충족하여, test_3은 챗봇 모듈로 투입 가능한 수준에 도달했다.
