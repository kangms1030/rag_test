# RAG3 (test_3) Phase 6 — 자율 분석·실험을 통한 고도화

너는 완성된 한국어 문서 RAG 파이프라인 **RAG3(test_3)** 를 인계받아 고도화하는 엔지니어다.
임무: 아래 사전 분석을 출발점으로 모델을 직접 분석·실험·조사하고, **검증된 개선만** 반영해
두 축(① 응답 속도, ② 다문서 종합·추론 대응력)에서 더 진보한 모델로 발전시켜라.

핵심 원칙 4가지:
1. **test_3의 틀을 벗어나지 마라** (§5에 "틀"을 명시함. 재설계·프레임워크 교체 금지)
2. **기존 코드는 실험 기간 내내 무수정** — 개선은 전부 병행 실험판(§8)에서 같은 형태로
   구현·측정하고, 기존 파이프라인은 언제든 그대로 돌아가는 대조군으로 보존한다.
   본 코드 반영 여부는 네가 아니라 **최종 A/B 비교표를 본 사용자가 결정**한다
   (대체 / 유지 / 병행 중 택일).
3. **측정 없는 주장 금지** — 모든 개선 효과는 동일 평가셋에서 기존판 vs 실험판 실측 비교로만 입증
4. **환각 0 · 무관질문 거절 100%는 어떤 개선으로도 후퇴 불가** (이 모델의 존재 이유)

지금은 plan 모드다. §1의 자료를 읽고 우선순위·실험 설계·검증 계획을 담은 계획을 먼저 제시하라.

## 2. 현재 모델 확정 사실 (재조사 불필요)

- 파이프라인: MinerU pipeline 파싱(`parsed_v25` 캐시, 13문서/969페이지) → 블록 청크 2,562개
  (**카탈로그 프리픽스 주입 — 리랭킹에 load-bearing, 포맷 변경 금지**) → embeddinggemma + BM25(kiwi)
  RRF 하이브리드 → bge-reranker-v2-m3 → small-to-big 페이지 승격 → gemma4:12b 답변 →
  검증(숫자 결정론 대조 + groundedness) → 결정론 롤백/CRAG (controller S1~S8)
- 성능(36문항, controller 경로): page_hit@3 **0.774** / doc_hit@3 **1.0** / **환각 0** /
  무관거절 **100%** / 평균 **63.4s** / 최대 148.6s / 평균 모델호출 2.97회
- 환경: 전 구간 로컬(Ollama, API 키 없음), RTX 5060 Ti 16GB, conda `intern_chatbot`, Windows.
  실행 규칙은 프로젝트 CLAUDE.md 준수 (`cmd /c conda activate intern_chatbot && ...`)
- 인계 계약: `Rag3Engine.ask()` 반환 스키마(README의 AskResult 표)는 다른 개발자가 LangGraph
  연동에 사용할 계약 — **기존 키 불변, 추가 키만 허용**

## 3. 사전 분석 결과 (직전 세션에서 실측 완료 — 그대로 활용하라)

### 3.1 속도 병목 분해 (varfix_eval.json 36문항 실측)

| 구간 | 근거 | 소요 |
|---|---|---|
| S1~S3 (임베딩+하이브리드 검색+리랭크) | 무관질문 5건 = LLM 답변 없이 이 구간만 실행 | **1.0~3.1초** |
| 12b 호출 1회 (답변 or 검증) | 2호출·재시도0 문항군 평균 23.5초 | 호출당 **10~15초** |
| length-retry 페널티 | 재시도0 평균 23.5s vs 재시도2 평균 **131.2s** | 문항당 **+100초** |
| 꼬리 (>90s, 11문항) | 전부 재시도 2회 소진 또는 5호출 abstain 경로 | 최대 148.6s |

**결론: 시간의 95%+가 "gemma4:12b 호출 시간 × 호출 수". 검색 스택은 병목이 아니다.**
- 호출수별 평균: 2회=47.0s / 3회=63.4s / 4회=91.4s / 5회=144.2s
- 최악 꼬리의 주범은 콜드 KV whitespace 런어웨이 방어용 length-retry (config 주석 참조)
- 함의: **로컬 12b 유지로는 "모든 답변 20초" 불가** (호출당 10~15s × 최소 2회).
  로컬 개선의 현실적 목표는 "정상 경로 대부분 30s, 최악 90s"까지.

### 3.2 종합·추론 질문이 약한 기전 (코드 근거 확인됨)

1. **검색이 단일문서 사실확인형에 캘리브레이션**: `top_docs 2 / final_pages 3 / retrieve_candidates 20`은
   전부 "정답이 한 문서 한 페이지" 가정의 Phase 1 최적값. 복합 질문은 단일 임베딩으로 "평균 방향"
   검색이 되고, 크로스인코더는 질문의 절반만 답하는 청크에 중간 점수를 줘 top3가 한 문서로 쏠림.
2. **검증 게이트가 회피를 유도**: groundedness 검증이 여러 문서를 연결·재구성한 문장을 미지원으로
   판정하면 **답변 전체가 abstain**. 관찰된 "회피 경향"의 직접 원인 (환각 0의 대가).
3. **12b 사고력 천장**: 다근거 대조·종합 능력 자체의 한계.
4. **측정 불가 상태**: 현 36문항 평가셋에 종합형 문항이 0개 — 개선 이전에 측정 수단부터 없음.

## 4. 개선 후보 백로그 (우선순위 제안 — 계획 수립 시 실측으로 조정 가능)

### P1. 속도 — 로컬 (승인 불필요, 즉시 착수 가능)

| 후보 | 내용 | 기대효과 / 위험 |
|---|---|---|
| (a) 꼬리 fail-fast | length-retry 2회 소진 시 후속 경로 생략하고 즉시 abstain. 대상 문항(vp_006/007, core_002/008 등)은 파싱 천장이라 어차피 abstain이 정답 (FINAL_REPORT §5-B) | 최대 148s→~60s, 정확도 손실 없음(가설) / 해당 문항 재실행으로 확인 |
| (b) 검증 조건부 스킵 | 숫자 대조 통과 + rerank top 고점수 + 단일 문서일 때 groundedness LLM 호출 생략. 차선: `verify_model: gemma4:e4b` 스왑 ablation (config에 스위치 기존재) | 정상군 23.5s→~15s / **환각 0 약화 위험 → 36문항 전체에서 환각 0 유지 필수 확인** |
| (c) 적응형 컨텍스트 트림 | 리랭크 점수가 top 대비 급락하는 페이지를 컨텍스트에서 제외해 prefill 절감 (`context_max_chars` 10000이 prefill 비용의 원흉) | 호출당 20~30% 단축(가설) / page_hit·kw 회귀 확인 |
| (d) 스트리밍 | 챗봇 체감용 — 답변 스트리밍 + 검증은 후행 배지. Rag3Engine에 콜백 추가는 **추가 키/인자로만** | 체감 지연 절반 이하 / 스키마 계약 유지 |

### P2. Gemini 2.5 Flash 백엔드 (**사용 승인 완료** — 이번 Phase의 핵심 실험 축)

사용자가 보안 검토를 마치고 외부 API 사용을 승인했다. **API 키는 프로젝트 루트 `.env`의
`GEMINI_API_KEY`** 에 있다. 모델은 **gemini-2.5-flash 하나만** 사용한다.

- **치환 범위**: 로컬 gemma4:12b가 담당하던 LLM 호출 자리 전부 — S6 답변 + S7 검증(groundedness)
  + CRAG judge, 그리고 P3의 분해·합성 호출. 즉 "현재 로컬 모델을 쓰는 자리에 그대로" Flash가
  들어가는 형태다. `models.py`의 백엔드 추상화(get_backend)와 동일한 인터페이스를 구현한
  Gemini 백엔드를 **실험판 쪽에** 만들어, 컨트롤러 로직 무수정으로 끼워지게 하라.
- **절대 금지: 임베딩·리랭커 교체** (임베딩 교체 = 2,562벡터 전체 재색인 + API 영구 종속.
  검색 품질 0.774는 로컬 스택의 성과이므로 불변). **검색은 로컬, 생성·검증만 Flash.**
- **검증할 가설 4개** (이 실험의 존재 이유):
  1. **속도** — 정상 경로 8~12s 예상. 콜드 KV whitespace 런어웨이는 Ollama+Gemma 아티팩트라
     현상 자체가 소멸 → retry 꼬리(최대 148.6s)가 통째로 사라지는지, "전 문항 20s 이내"가
     실제 달성되는지
  2. **정확도** — avg_kw_hit 0.528(12b 기준)이 개선되는지
  3. **사고력 천장** — P3 종합형 평가셋에서 12b 대비 격차가 실제로 벌어지는지
  4. **(확장) 파싱 천장** — Flash는 멀티모달이므로, 기존 vision 경로(vision_answer_model 자리)에
     Flash를 끼우면 근거존재-abstain 5문항(vp_006/007/013, core_002/008 — FINAL_REPORT §5-B)이
     열리는지. 유일한 미달 지표(근거존재-실패율 20.8%)를 공략하는 저비용 실험
- **API 운영 규칙**:
  - 키는 `.env`에서만 로드(python-dotenv 또는 직접 파싱). **키 값을 코드에 하드코딩하거나
    로그·보고서·콘솔·git에 노출하는 것 절대 금지.**
  - 온도 0(결정론 지향). Gemma 전용 방어(retry_on_length, num_predict 튜닝 등)는 Gemini
    경로에서 비활성화하고, 대신 API 타임아웃 · 429/5xx 지수 백오프 재시도 · 네트워크 장애 시
    **로컬 12b 폴백**을 설계할 것 (전 구간 로컬로 돌아가는 길은 항상 열어 둔다)
  - 호출마다 입출력 토큰 수를 기록해 **질문당 비용을 실측**하고 최종 비교표에 포함하라.
    무료 티어라면 RPM/일일 한도를 먼저 확인하고 평가 실행 페이스를 그에 맞출 것

### P3. 다문서 종합·추론

| 순서 | 내용 |
|---|---|
| (a) **평가셋 먼저** | 13문서 코퍼스에서 종합형 10~15문항 초안 작성(문서 간 비교/공통/집계형) → **사용자에게 정답 검수 요청** (사람 검수 없이 개선 착수 금지) |

"복합 질문에서만" 발동하도록 라우팅으로 격리할 것 (단일형 질문 경로 무영향).

## 5. 유지해야 할 "틀" (위반 금지 — 이 목록이 test_3의 정체성이다)

1. **아키텍처 불변**: MinerU 파싱 캐시(parsed_v25) / 블록 청크 + 카탈로그 프리픽스(포맷 불변) /
   flat numpy + BM25 RRF / bge-reranker-v2-m3 / small-to-big / controller 상태기계 / Ollama 로컬 기본
2. **디스크 포맷 불변**: `vectors.npz` + `docs.json` + `page_store.json`
3. **API 계약 불변**: `Rag3Engine.ask()` 반환은 기존 키 유지 + 추가만 (웹데모·인계 문서가 의존)
4. **교체 금지**: 임베딩 모델, 벡터 저장 방식, LangChain/LlamaIndex 등 프레임워크 재작성
5. 실험판의 신규 동작은 전부 설정으로 on/off 가능하게 — **off면 기존판과 완전 동일하게
   동작해야 하며**, 이것이 곧 사용자가 "병행"을 선택할 경우의 운영 스위치가 된다
6. **기존 파일 무수정 원칙(엄격)**: rag3 패키지·ask_cli·webapp의 기존 파일은 실험 기간에
   한 줄도 수정하지 마라. 모든 구현은 §8의 병행 실험판에서만. 기존 파일 수정이 정말
   불가피하다고 판단되면, 수정 전에 사유와 diff 계획을 사용자에게 보이고 확인받을 것
7. Phase 4/5 검증 결과(FINAL_REPORT, PHASE5 §3)를 깨는 어떤 변경도 금지

## 6. 검증 규칙 (모든 변경에 적용)

1. 변경 전 golden 확보: `test_3/tmp/verify_controller_regression.py` 실행해 기준 확인 (~4분)
2. 각 실험 채택 판정은 **controller 경로**로만: 수시 체크는 위 스크립트, 최종 판정은
   루트 `tmp/eval_varfix.py` 36문항 전체 (~40분, GPU 단독 상태에서)
3. **`python -m rag3 evaluate`/`ask`는 단일패스(generate_answer) 경로 — FINAL 수치와 비교 금지.**
   검색 지표(doc/page_hit) 확인용으로만 사용 (README에 명시된 함정)
4. 속도 실험은 반드시 **웜 상태 기준으로 비교** (콜드 편차가 크다. 워밍업 후 측정)
5. 절대 지표: **환각 0, 무관거절 100%** — 하나라도 후퇴하면 해당 개선은 즉시 미채택

## 7. 운영 함정 (전부 실측된 사고 — PHASE5_DEPLOYMENT_REPORT §4)

- **GPU 프로세스 단독 실행**: 평가/서버/add 중 2개가 겹치면 둘 다 **로그 없이 무증상 크래시**
- **백그라운드 작업 알림을 믿지 마라**: 이 프로젝트에서 완료 알림과 실제 프로세스 상태가 어긋난

- 신규 문서 색인은 반드시 `python -m rag3 add` (풀 ingest는 parsed_v25 폴백 함정으로 청크 0개)

## 8. 진행 방식 — 병행 실험판과 최종 A/B 비교

### 8.1 병행 실험판 구성 (기존 코드 무수정의 실현 방법)

- 기존 rag3는 손대지 말고, **별도 실험 패키지**(예: `test_3/rag3x/`)를 만들어 그 안에서
  개선판을 구현하라. 구성 원칙:
  - 동작이 바뀌지 않는 부분(설정 로드, flat 인덱스, 검색, 리랭커, page_store 등)은 기존 모듈을
    **import로 재사용** — 코드 복제 금지
  - 동작이 바뀌는 부분(controller/answer/models의 변형, Gemini 백엔드)만 실험 패키지에
    명시적 포크로 두고, 파일 상단에 원본 경로와 변경 요지를 주석으로 남겨라
  - 실험판은 인덱스·파싱 캐시·카탈로그를 **읽기 전용**으로만 사용 (공유 자원 변형 금지).
    실험 산출물은 `test_3/tmp/`와 `probes/results/`에만 기록
- 실험 도중 수시로 `verify_controller_regression.py`를 돌려 **기존판 무손상**을 계속 증명하라

### 8.2 사용자 확인 게이트 (이것 외에는 자율 진행)

1. **기존 파일 수정이 불가피할 때** — 사유 + diff 계획을 먼저 보이고 확인
2. **인덱스/캐시/카탈로그를 파괴할 수 있는 작업**
3. **종합형 평가셋 정답 검수(P3-a)** — 초안을 만들어 사용자 검수 요청

### 8.3 최종 산출물 — A/B 비교 보고서 (이 작업의 종착점)

모든 실험이 끝나면 `test_3/probes/results/PHASE6_COMPARISON.md`에 **동일 조건 비교표**를 작성하라:
